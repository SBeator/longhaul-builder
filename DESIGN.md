# longhaul-builder — 设计文档（单一事实源 / SSOT）

> **一句话进 → 一个 AI 自主、长时间地跑 → 一个中等规模（按周计）的项目被建出来。**
>
> 这里真正的交付物是**这套流程 / 能力本身**：一个能**长时间自主运转、又不跑偏**的构建 loop。它**不绑定任何具体 agent、不绑定任何具体基础设施，能搬到别人的机器上跑**。它顺手做出来的那个项目，只是引擎的一个*产物*——不是引擎本身。

> `longhaul-builder` 只是个占位名，随意改。
> 本文件是单一事实源。要打磨设计，就直接改这个文件，别在别处重新争论已经定下来的决定。

---

## 0. 它是什么 / 不是什么

- **它本身** = 一个**不绑定 agent、不绑定基础设施的「长跑自主构建 loop」**。它的全部价值就在于能**长时间自主运转而不跑偏**。
- **核心判据**：如果它只能跑一小阵，那就**毫无价值**——短时间的自主写代码，现成的框架 + 一个够强的模型早就解决了。所以「能无人值守、长时间、不跑偏地跑」*就是它的全部意义*，不能砍。
- **不做什么（non-goals）**：
  1. 不追「一周全自动搞定」那种 demo 级神话。这件事真实世界的成功率很低（Devin 实测大约 15%）。我们是在前沿地带**搭结构化的护栏**，不是去抄一个已经成型的产品。
  2. 不耦合任何具体 agent、任何具体基础设施（见 §2.1 依赖契约）。
  3. MVP 不做多模型适配。

## 1. 设计立场（最底层的几条——下面所有东西都从这里推出来）

1. **所有状态都活在 agent 之外。** agent 的上下文只是草稿纸，项目的真相活在文件里。崩溃可续、换 agent 不用重启、独立 reviewer 这三件事，全是从这一条规则里自然掉出来的。
2. **反着用「超大上下文窗口」这个直觉。** 长上下文是*病*，不是药：误差会复利累积（每步 99% × 100 步 ≈ 37% 存活），而且目标会漂移——agent 容易被它最近一次 tool 输出牵着走，或者过早宣布「做完了」。所以 **driver 是短命的**：每次被唤醒只啃一小片，然后**死掉**。连续性靠外部进度账本 + 外部 scheduler 再次唤醒它来保证。**永远别赌「它会记得」。**
3. **拒绝提早收工——但永远留一个熔断器。** 外层 loop 推着 agent 别假装完成、继续往前。但每次重新唤醒前都先看 `attempt_count`：撞到上限、或者局面根本不收敛，就把这个 milestone 标成 `BLOCKED`、上报给人。绝不无限重启白烧 token（前车之鉴：Devin 朝一个不可能的方向死磕了好几天）。
4. **验证是内建的，每一步都留下证据——而且证据由确定性脚本产出，绝不是 agent 自述。** 这堵住了 reward-hacking：如果被审计的那条 trace 是「被审计的人」自己写的，它完全可以跳步，再写一段能糊弄 judge 的话。
5. **人在 loop 里，但很少出场。** 只有 `P0` 决策（不可逆 / 方向性错误）才硬停等人。`P1` 那种可逆的假设，由 AI 自己拍板并记进假设账本。`P2` 进风险清单，不阻塞。

## 2. 第一层 —— 能力本身（不绑定 agent / 基础设施 —— **这就是交付物**）

### 2.1 依赖契约（硬约束：只依赖这几样浅的、可移植的东西）

> 目标：**它必须能搬到别人的机器上跑。** 所以依赖面必须薄到极致。

- **一个能读写文件、能跑 shell 的 coding agent。** 同一个 agent 本体既当 *driver*，也当 *reviewer*。
- **一个带版本的状态存储**：一个 git 仓 + 一套目录约定。
- **一个能定时 / 反复唤醒东西的 scheduler**：cron、CI 的定时任务、一个前台守护的 `while` 循环、或者 `flock`——任意一种就够。

**明确不依赖什么：** 第一层里没有任何东西依赖某个聊天工具、通知通道、看板产品、或某套具体基础设施。那些全是第二层的事（见 §3）。

### 2.2 状态账本的结构（外化的真相 + 程序计数器）

账本活在**被构建的那个项目里**，放在一个隐藏的 `.longhaul/` 目录下——和 `.git` 一样，状态跟着项目走，而不是寄居在框架里（见 §2.7）。`.longhaul/` 里面：

| 文件 | 作用 |
| --- | --- |
| `spec.md` | 冻结的需求（验收探针、P0/P1/P2、假设账本） |
| `milestones.json` | milestone 列表；每条 `{id, goal, acceptance（验收方式 + probe）, status: TODO/IN_PROGRESS/DONE/BLOCKED/SKIPPED, attempt_count, max_attempts（默认 3）, last_error}` |
| `cursor.json` | **程序计数器**：当前 milestone、当前 task、**下一步要做的那个确切动作**。每次启动都确定性地接着*这一步*往下走。**不许自由重新规划**——自由重规划是漂移的源头。 |
| `events.jsonl` | 只追加的决策 + 事件流（审计和回放都从这里来） |
| `evidence/` | 每一步的验收证据（原始测试输出、截图、golden output、checksum……），**由确定性脚本写入** |
| `handoff/` | 每个 milestone 边界处压缩后的状态 |
| 每步一个 git commit | 一个可回滚的检查点 |

### 2.3 五阶段流程

1. **spec 熟成（spec-driven）。** 一句话 → 苏格拉底式追问 + 分块确认 → `spec.md`。字段：User Stories / 验收标准（可测、可量化）/ **每条标准对应一个验收 probe**（说清这条怎么验）/ `[NEEDS CLARIFICATION]` / 假设（P1，可逆，AI 拍板）/ Decision Log / 风险（P2）。熟成度门：**P0 必须为空 = 硬 gate**（放行前要人确认）；P1 由 AI 自己定并记下来、不阻塞；P2 进风险清单。
2. **milestone 拆解。** 拆成**可独立验证**的若干 milestone，每条都带它的验收方式 + probe；写进 `milestones.json`。
3. **自主长跑推进（最难的部分）。** 外部 scheduler 循环：读 `cursor.json` → 派一个**短命 driver** 去啃当前 task（§2.4）→ 写证据 + 推进 cursor → driver **死掉** → 下一个 tick 用一份*全新*的上下文接着跑。每跨一次 milestone 边界，都要走一次 handoff 压缩。
   - **每个 milestone = 两道门、两次产出、两次审查**（为的是把返工成本压下来）：
     1. **先出方案。** driver 先只产出*一份方案*（思路 / 设计 / 测试策略 / 改哪些文件）——**还不写代码**。
     2. **审方案（Gate 1）。** 一个独立 reviewer 审*方案本身*：靠不靠谱？有没有更好的路子？跟 spec 对不对得上？不过 → 打回去重出方案（计一次 attempt；撞上限就熔断）。过 → 放行去实现。
     3. **实现。** driver 照着*已商定的方案*用 TDD（红 → 绿）来建，并产出真实证据。
     4. **审实现（Gate 2）。** 一个独立 reviewer 确认测试是真的过了（证据是真的，reviewer 自己重跑一遍）**外加一次代码质量审查**（三层裁定，见 §2.4）。
     5. 两道门都过 → `complete`，推进 cursor。
   - **失败路由（省返工）：**
     - **Gate 1（方案）不过** → 回到 **① 出方案**，带上不过的理由重写。（方案没定下来之前，不碰代码。）
     - **Gate 2（实现）不过** → **默认：回到 ③ 实现**，带上 reviewer 的理由，*继续修实现*。**不**重开方案（方案在 Gate 1 已经定死了）。
     - **逃生口：** 只有当 Gate 2 的 reviewer 明确裁定「**是方案本身错了，不是实现问题**」时，才回到 ① 重做方案（少见）。
     - **熔断器：** 如果实现重试的累计 `attempt_count` 超过 `max_attempts` → 标 `BLOCKED` 并上报。不无限烧。
   - **为什么要两道门：** 如果一个错方案要等到实现完才发现，返工就贵了。先把方案定下来，能在写任何代码*之前*就把「方向错了」挡在门外。Gate 1（审方案）轻快；Gate 2（审实现）重，要有证据撑。
4. **双角色质量门。** 见 §2.5。
5. **熔断 + 交付。** 见 §1 立场 3。所有 milestone 都 `DONE` + 最终验收 → 交付。

### 2.4 单 task 的实现 + 验证（质量地板）

- **TDD 优先**（凡是合适的地方都上 TDD）。理由：这套能力是给那些**本来根本不会写测试**的人用的，所以 TDD 是**质量地板**，不能交给用户自己看着办。流程：先写测试，**看着它变红** → 实现到**变绿**。
  - **反作弊护栏**（「你没法靠 prompt 哄出 TDD 纪律」）：写测试和写实现是**读写隔离**的（两个互相看不见的 subagent）；每一次红/绿切换都必须贴出**真实的原始运行输出**——不许靠推断或者幻觉「它过了」。
- **TDD 不合适的地方**（探索性 / 纯 UI / 难以表达成用例）：别硬上；退回到下面的可观察验收门 + 一条日志旁路。
- **通用质量地板（每个 milestone 强制，叠在 TDD 之上）= 这个 milestone 的「可观察行为验收门」。** 按项目类型挑验收方式——Web = 真浏览器访问 + 截图 / CLI = golden output diff / API = 契约测试 / 数据 = fixture + checksum / bot = dry-run + 事件 + 回放。留证据 → 不过就把证据喂回 driver、修到绿。再配一条**日志定位旁路**（对付难缠 bug：埋点 → 复现 → dump 日志 → 分析）。

- **验收裁定 = 每个 milestone 都过自己的门，独立按证据裁定，绝不信 driver 自述：**
  - **每个 milestone 在 complete 之前都必须过自己的验收门**（如果它的门里含 E2E，就真跑 E2E）——**不是**「到最后才跑一次 E2E」。过了才 `complete`。
  - **裁定分三层**（三层全过才能 `complete`）：
    - ① 一个确定性脚本**真的去跑** probe，并抓到**真实证据**（exit code / 原始输出 / 截图 / 日志）。
    - ② **证据是真的 + probe 过了。** driver **不许自报通过**：客观 probe（TDD / golden / 契约）由脚本的**真实 exit code** 裁定；**E2E / 视觉**类检查由一个独立 reviewer 读证据来裁定。「driver 说 E2E 过了」不算数。
    - ③ **对方案 + 代码本身的质量审查。** 独立 reviewer **必须真的去审**这一步的*思路*和*代码*：思路合不合理、有没有更好的、跟 spec 对不对得上、代码在结构 / 边界 / 坏味道上对不对？**「测试全绿」≠「过了」：** 哪怕每条自动化测试都绿了，只要方案或代码不合理，照样打回。（这是在防它退化成一个只会查「测试过没过」、从不审设计的窄 judge。）
  - 最后那次「全链路 E2E」是把**所有** milestone 拼在一起再端到端跑一遍（一道额外的集成检查），它**不是**唯一的 E2E。

### 2.5 两个角色（driver + reviewer）

- **driver** 是**唯一的写入者**，短命，每个 tick 啃一片。
- **reviewer / critic** 是**只读、上下文隔离、角色分离**的。它拿**两把尺**量（*两把都过才放行*）：
  - **(a) 质量 / 正确性 / 设计：** *真的去审*这一步的思路和代码——思路合理吗？有没有更好的选项？跟 spec 对得上吗？正确性 / 结构 / 边界 / 坏味道？**测试全绿 ≠ 过：** 方案或代码不合理照样打回。
  - **(b) 流程合规 + 证据真实性：** spec 熟成并冻结了吗？milestone 带验收吗？是先红后绿吗？验收 probe 真的跑了吗？证据是真实运行的产物、没有跳步作弊吗？
  - 用 **Agent-as-a-Judge** 的模式，对着一条 *trace* 来审。
  - **反作弊的关键点：** 审计的输入（trace / 证据）**必须由确定性脚本从一次真实运行里产出**，不能是 driver 嘴说的——否则一个被跳过的步骤完全能被写成「合规」、把 judge 糊弄过去。
- **并发设定：** 多个写入者实时同时改 = **禁止**（上下文漂移 + 错误连锁）。允许的是：一个写入者 + 一个只读 critic（critic 可以在旁路里实时审）；task 级并行 = 各自在隔离的 worktree 里跑，最后按文件 / PR 合并。
- **看门狗（watchdog）**（卡死检测 + 再唤醒）= 工程界标准那一套四件套 `lease + heartbeat + TTL 扫描 + rerun`。它**不**起一个常驻的监控 agent，并且**它来强制执行熔断器**（超过 `max_attempts` → BLOCKED + 上报）。

### 2.6 随时可介入的通道（inbox）

人在 loop 里，不只是那两道强制停下来的门——还有一条**可选的、随时能用**的介入通道，它是「状态外置 + loop 每个 tick 重新读文件」天然带来的。人不必去打断一个正在思考的 AI；他丢一条消息进去，下一轮 loop 迭代就自动把它吸收进来。

- **概念澄清：** 编排器是一个**极小的确定性 loop**（不是 AI），不是「一个写死了剧本的大脚本」。每个 tick 它只做：读 cursor → 派 AI 跑一步 → 记证据 → 推进 / 重试 / 熔断。*做什么、做几步*来自**状态文件**（在 AI 规划阶段产出的），不是来自脚本。
- **第一层的机制：** 一个运行目录里放一个 `inbox/`（或 `inbox.jsonl`）。loop **在每个 tick 开头消费 inbox**：
  - `respec`（改需求 / 改计划）→ 路由到「把受影响的 spec / milestone 重新熟成」，更新文件，必要时触发一个 P0 小门让人确认，然后从**新计划**接着跑。
  - `redirect`（改某个 milestone 的思路）→ 更新这个 milestone 的指令并把它重新排队。
  - `pause` / `resume` / `abort` → 控制 loop 本身。
- **为什么这是「白送」的：** 一个短命 driver 每次全新启动都会读*最新*的 spec / milestones，所以人的改动是**持久化的、被下一步自动接住**的——根本不用去「逮」一个正在跑的 agent。
- **别把两种「人在 loop」混为一谈：** 强制等待 = **2 个点**（P0 / 交付验收）；可选介入 = **随时**（inbox——想管就管，不管它也自己跑）。
- **第二层绑定：** inbox 的投递口可以接到一个聊天工具 / 通知通道上（你发一条消息 → 它被写进 inbox）。第一层本身只认文件，不绑任何聊天工具。

### 2.7 项目布局 + 跨项目跟进（registry）

**铁律：框架不是项目的父目录。** 每个被构建出来的项目都是一个**独立、平级**的仓，有自己完整的目录结构；框架只提供那台*机器*。

- **框架** `longhaul-builder/` = 纯机器（状态 / loop / prompts）。它**不存任何项目状态**。
- **一个被构建的项目** = 一个独立的平级仓（比如 `<你的项目>/`）；它的构建账本就内嵌成它自己的 `.longhaul/`——和 `.git` 一样，跟着项目走、自包含、可移植。
- **跨项目跟进（一种能力）：** 项目们平级散落在各处；用一条 **registry + 看板命令**给出统一视图：
  - 引擎新建一个项目时，自动注册进 `~/.longhaul/registry.json`（只记 name + path；放在用户级，这样框架保持纯代码）。
  - `lhb status`（或 `registry.py status`）遍历 registry → 读每个项目的 `.longhaul/cursor.json` + milestone 计数 → 打印一张跨项目看板：谁在哪个阶段 / 走到哪了 / 谁 BLOCKED 卡在等人。
  - **多个项目并行** = N 个独立 loop（各跑各的节奏、各有各的 `.longhaul/`）+ 这一张聚合视图；互不干扰。

### 2.8 carry-forward，正式化（reviewer 的不阻塞小意见 → 喂给下一个 milestone）

reviewer 的裁定不只有「过 / 打回」——还有第三种：**过了，但带几条不阻塞的小意见 / 条件**（`PASS_WITH_NITS` / `APPROVE_WITH_CONDITIONS`）。这些小意见不该把当前这一步打回（那会卡住进度），也不该被丢掉（下一个 milestone 可能正好踩到它）。**carry-forward = 把它们提炼成跨 milestone 的 carry 项**，让下一个 driver 在规划 / 实现之前先读到它，把「上一步发现了但当下不必返工的事」当成自己的输入。这是 §2.4 验收门的自然延伸，也是「短命 driver 不积累长上下文」和「经验不丢失」之间的那座桥。

- **单一事实源** = `.longhaul/notes.md`。格式（机器可 grep、人可读）：
  `## carry-forward (<milestone> · <kind> · <time>)\n> <text>`。
- **两条写入路径，同一种格式：**
  1. **自动：** reviewer 返回 `PASS_WITH_NITS` / `APPROVE_WITH_CONDITIONS` 时（Gate 1 审方案和 Gate 2 审实现都能触发），loop 把 carry 项持久化下来，并记一条 `carry_forward` 事件。
  2. **手动：** `state.py note <state_dir> <milestone> "<text>"`（人 / 脚本把一条小意见显式记进 notes.md）。
- **消费：** 下一个 milestone 的 driver prompt 里有一个 `{{carry_forward}}` 占位符指向 notes.md；「规划 / 实现之前先读这个文件」是写在 notes.md 顶部的硬约定。`redirect`（§2.6）用到的那个 per-milestone `note` 字段，走的是同一条渲染管线。
- **它在流程里的地位：** 这是**一等公民**，不是事后补丁——它让「每步一个独立 reviewer」的产出能积累起来，而不必依赖某个长命 agent 去「记住上一步说了啥」。

### 2.9 门的强度按风险定

**不是每个 milestone 都配得上两道全门。** 两道门（出方案 → 审方案 → 实现 → 审实现）是为**核心 / 有风险**的步骤设计的——这种步骤一旦方向错了，返工很贵。对**机械 / 脚手架**类的步骤（搭脚手架、纯文档、确定性的模板生成），方案几乎没有「方向风险」，跑两道门纯属走过场。所以：**给每个 milestone 按风险分配门的强度。**

- **门强度提示（`gate`）：** 挂在 milestone 上（`milestones.json` 里一个可选的 `gate` 字段，或者在 spec 的熟成度门里标注）。取值：
  1. `light`（机械 / 脚手架 / 纯文档）：**一道轻门**——规划可以并进实现，过一次证据 / 质量审查（Gate 2）即可。
  2. `full`（核心 / 有风险 / 方向不可逆）：**两道门**（默认）——先审方案定方向，再审实现。
- **谁来定：** 熟成 / 规划阶段（age/plan + 人）根据「这步错了返工贵不贵、是不是真有方向上的选择」给每步定 `gate` 提示。**编排器 / loop 读这个提示来决定跑几道门**，它不在运行时临时拍板。默认 = `full`（保守——宁可多审一遍）。
- **本项目拿自己当 dogfood：** 机械类特性走 `light`；核心机器类特性（状态机 / 证据门 / judge / 自驱 loop / inbox / watchdog）走 `full`（两道门）。
- **诚实的 MVP 现状：** `loop.py` 目前对所有 milestone 都跑同一套两门转换；`gate` 提示的**自动路由**（让 `light` 步骤跳过 Gate 1）已经设计好、**等接线**——提示放哪、语义是什么本节都定义清楚了，「loop 读它来路由」是下一个小步骤（它不影响已经验证过的 `full` 路径）。

## 3. 第二层 —— 可选的环境适配层（**延后**）

第二层是一个**可选的环境适配层**：把第一层里的每个能力件，换成*你自己环境里*那套已经硬化的实现——一个 scheduler、一个 judge、一个看板、一个通知通道、一个调研步骤、一个「构建并自测」的工具——**而核心一行都不用改。**

原则：框架那个纯核心，可以由你环境里硬化的适配器在背后撑着。**唯一一条红线**：绑定 / 适配层**绝不能重新实现**能力层——它只适配入口和执行，不重造那台机器。把每个适配器接成一层薄薄的转接片，去调你已有的原子能力；别把整个引擎揉进一个大 prompt 里。

> 工作顺序：先把不绑基础设施的核心建好，再去绑硬化的适配器。第二层是后续工作。

## 4. 可移植性（在别人的机器上跑起来）

因为第一层只依赖几样浅的东西——**一个 coding agent + git + cron**——所以它是可移植的：配一份 INSTALL 指南打成包，就能在另一台机器上跑起来。

四样最小依赖（大多数人本来就有 cron + git + 一个 coding agent）：

1. 一个 coding agent（任意：Claude Code / Codex / Cursor……）。
2. git + 那套目录约定（`spec.md` / `milestones.json` / `cursor.json` / `evidence/`）。
3. cron + `flock` / 一个 CI 定时任务 / 一个前台 `while` 循环。
4. 三套 prompt / rubric（熟成访谈 / 驱动执行 / 双轴审查）。

| 能力 | 可移植的替身 | 硬化版（第二层） |
| --- | --- | --- |
| 状态 | git + 目录约定 | 一个硬化的状态账本 |
| 熟成 | 一套访谈 prompt 模板 | 一个调研步骤 + spec 模板 |
| 检查点 | 在边界处写 `handoff.md` | 一个 handoff 辅助器 |
| 执行 | TDD/E2E + 证据文件 | 一个「构建并自测」的工具 |
| 审查 | 第二个 agent 调用 + rubric JSON | 一个硬化的 judge |
| 进度 | cron + flock / while 循环 | 一个硬化的 scheduler |

**终态** = 一个打好的包（文件 + INSTALL.md + 这份 DESIGN.md 当移植指南 + 占位符）；有了上面那四样，任何人都能跑起来。

## 5. MVP 定义（长跑自主 = 砍不掉的核心）

- **成功判据** = 引擎能在**无人值守、长时间、跨多次上下文重启**的条件下，自主把一个**多 milestone** 的项目推到完成；它能**自恢复**（从账本续跑 / 卡住就熔断 + 上报）、并且**不漂移、不遗忘**。证据 = 一份完整的 `events.jsonl` + `evidence/` + 真实验收 + 一份**无人工介入记录**。
- **可砍（保持小）：** 测试项目的复杂度 / 特性数；多模型适配；实时 critic 订阅（开头先做一次最终审查就够）。
- **砍不掉：** ① 长跑自主本身（短阵子毫无价值）；② 外部账本 + 程序计数器；③ 熔断器；④ 验收门 + 确定性证据。
- **测试靶子：** 一个 **Avalon（阿瓦隆桌游）发牌 web app**——多个玩家登录，按桌游 Avalon 的严格规则发角色，每个玩家的页面只显示自己的角色，外加按条件给的额外信息（比如 Merlin 能看到坏人是谁）。它被建成一个**独立平级仓**，状态内嵌在 `.longhaul/` 里（自包含）。选它是因为：后端的发牌 / 可见性算法客观可 TDD，前端可以走浏览器 E2E，而且它是多 milestone、强积累——能把两道质量门和「不漂移」都逼到极限。更大的真实项目，留到引擎跑通、适配器绑好（第二层）之后再说。

---

## 6. 一路带着往前走的待解风险（建的时候盯着这几条）

1. **熔断器的精确定义：** `attempt_count` 什么时候 +1，什么算「一次 attempt 中途被打断」，以及「能收敛 vs 环境性死锁」怎么分——都需要一个**可执行的判据**（不能只是文字描述）。
2. **验收证据反作弊：** 由确定性脚本产出；agent 不许自报通过。
3. **模型无关协议尚未验证：** MVP 先在单 agent 上跑通；多模型适配延后（它是最关键的新假设，但先证明单 agent 能跑通）。
4. **按周计的长周期演进，业界没有成熟答案：** 现有 benchmark 只测孤立的 issue 修复；这件事是在前沿搭护栏（一个「拿现实校准」的风险）。

## 7. Roadmap（建了什么 / 还剩什么）+ 第二层绑定

> 自举构建（引擎用它自己那套两门 loop 来建这些特性）这边，第一层机器已经完整：F1–F8 全部 DONE，每个都带测试 + 证据；而且 `loop.py` 自驱跑通了一个真实步骤，留下了自举证据（见 `.longhaul/evidence/F8/selfhost-demo/`）。下面是「已建 vs 待硬化」的全貌。

### 已建（第一层本身，不绑 agent / 基础设施——这就是交付物）

| # | 特性 | 在哪 | 证据 |
| --- | --- | --- | --- |
| F1 | prompt / rubric 模板以文件形式存在 + 渲染器 | `engine/prompts/` + `prompts.py` | test_prompts |
| F2 | 状态机、两道门 + 精确 attempt 计数 + 熔断器 + 向后兼容 | `state.py` | test_state(_phases) |
| F3 | `verify.py` 确定性证据门（只按真实 exit code 裁定；反作弊） | `verify.py` | test_verify |
| F4 | `review.py` 独立 judge 适配器（agent 可换、对畸形输出能稳健降级） | `review.py` | test_review |
| F5 | `loop.py` 自驱 tick + `loop.sh`（cron / flock 包装） | `loop.py` `loop.sh` | test_loop |

### 待硬化（待办 / 第二层 —— **延后**）

1. **真 LLM 的 driver / judge 绑定（最关键的下一步）：** driver / judge 目前是一个「可配置命令」的抽象，stub 先行（用确定性命令跑 demo / 测试，不烧 LLM）。把命令换成一个真 coding agent（比如 `claude -p < {prompt_file}`）就能真正自主——**机器一行都不用改**（§2.4 那层命令抽象已经就位）。这正对应「模型无关协议尚未验证」那条风险。
2. **age / plan 从骨架 → 智能：** 今天 `age.py` / `plan.py` 是诚实的**骨架生成器**（标准分节的 spec + AC→milestone 1:1 stub）；真正的熟成（苏格拉底式追问 + 上网查规则 + 分块确认）和真正的拆解（依赖排序 + 粒度合并 + 补 `probe_cmd`）是第二层的活（一个调研步骤 + 一个真 LLM driver）。
3. **门强度自动路由接线**（§2.9）：`gate` 提示的语义已经定义好；让 loop 读它、放 `light` 步骤跳过 Gate 1，是一个还没接的小接线步骤。
4. **异步 worker：** 当前是同步 tick（一个 tick 把一个 phase 跑完再退出）；一个长步骤里没及时刷新的 heartbeat，靠 TTL 的宽限兜着。真正的异步 worker + 实时 heartbeat 是后续硬化。
5. **多模型适配 / 实时 critic 订阅：** MVP 明确不做。
6. **F5/F6/F7 带出来的不阻塞小意见**（见 `.longhaul/notes.md` 的 carry-forward）：
   - F5→F8：`step_timing` 给 driver/verify/review 这几个子步骤报的是 `rc=None`（结果是个 tuple，不是 int），所以时间线按时长着色、不按 rc（已采纳：这条时间线只用时长）；`state:claim` 子步骤把 phase 记成了 `"?"`（无伤大雅；时间线上显示 `(claim)`）。
   - F5→F8：`_phase_impl_review` 里那个 `vexit==2` 的死分支（`_verify` 只会返回 0/1）——可以删掉它 + 修一下 docstring；不阻塞。
7. **绑硬化的环境适配器（第二层，见 §3）：** 一个状态账本 / scheduler / judge / 调研步骤 / 「构建并自测」的工具 / 跨项目看板。**红线：** 绑定层不许重新实现能力层。
8. **跨项目 registry / `lhb status`**（§2.7）：第一层那套通用 registry + 聚合看板；把它绑到一个硬化的看板上，是后续工作。
